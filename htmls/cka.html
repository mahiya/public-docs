2025-2026 で 新規 / 強化 された出題ポイント

  • Helm: --set crds.install=false での CRD スキップ install (Argo CD)、helm upgrade --reuse-values、helm rollback <rel> <rev>。
  • Kustomize: 既存ディレクトリへの kubectl apply -k、overlay へ images: / replicas: 追記、commonLabels。
  • Gateway API: GatewayClass / Gateway / HTTPRoute、TLS termination、weighted traffic split、cross-namespace 経由の ReferenceGrant、Ingress からの migration。
  • Native sidecar: initContainers + restartPolicy: Always。既存 Deployment への logger sidecar 追加。
  • CRD / Operator: cert-manager / Prometheus Operator / Argo CD の install、kubectl explain でのスキーマ抽出、CRD 一覧の grep。
  • CNI / CRI / CSI: Calico Operator / Cilium CLI install、cri-dockerd .deb pkg、CSI driver 列挙。
  • In-place Pod Resize (v1.35 GA): kubectl patch pod ... --subresource resize。
  • HPA behavior: autoscaling/v2 + scaleDown.stabilizationWindowSeconds。
  • NetworkPolicy 強化: AND/OR 構造、kube-dns 例外、CNI 依存。
非推奨 / 弱められた領域

D1-01 RBAC: ServiceAccount + ClusterRole + 名前空間限定 RoleBinding Cluster Arch ★★☆ 5-10 分
namespace app-team1 に SA cicd-token を作成。ClusterRole deployment-clusterrole (verb: create, resource: Deployments,StatefulSets,DaemonSets) を作成し、app-team1 内でのみそのCRをSAに付与。kubectl auth can-i で検証せよ。 
kubectl create sa cicd-token -n app-team1
kubectl create clusterrole deployment-clusterrole \
  --verb=create --resource=deployments,statefulsets,daemonsets
kubectl create rolebinding deployment-rolebinding \
  --clusterrole=deployment-clusterrole \
  --serviceaccount=app-team1:cicd-token -n app-team1
kubectl auth can-i create deployments \
  --as=system:serviceaccount:app-team1:cicd-token -n app-team1   # yes
kubectl auth can-i create deployments \
  --as=system:serviceaccount:app-team1:cicd-token -n default     # no
落とし穴: ClusterRoleBinding にしないこと。--serviceaccount=ns:sa はコロン区切り (slash NG)。

D1-04 Aggregated ClusterRole (view への自動マージ) Cluster Arch ★★★ 8-12 分

D1-06 新規ノード Pre-Install (swap / br_netfilter / ip_forward) Cluster Arch ★★☆ 5-8 分

net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1

D1-14 etcd snapshot save Cluster Arch ★★☆ 5-8 分
/opt/etcd-backup.db に snapshot を取得し etcdctl snapshot status で検証せよ。

  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \

D1-15 etcd snapshot restore Cluster Arch ★★★ 10-15 分
/opt/etcd-backup.db から新データ dir /var/lib/etcd-restored に復元し、etcd static pod を再構成せよ。
# 1. 制御プレーン停止
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# 2. 復元
sudo ETCDCTL_API=3 etcdctl snapshot restore /opt/etcd-backup.db \
  --data-dir=/var/lib/etcd-restored

# 3. /tmp/etcd.yaml の hostPath を /var/lib/etcd-restored に編集

# 4. 戻す
sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/

D1 よくある落とし穴 (gotchas)

  1. SSH/context 切替忘れ — そのタスクは即ゼロ。
  2. RBAC subject 形式の混同: --as=system:serviceaccount:ns:sa vs --serviceaccount=ns:sa。
  3. cluster-admin の濫用 — 採点は特定 Role/Binding をチェックするため、過剰権限は不正解。
  4. Calico custom-resources.yaml の CIDR 編集忘れ — 192.168.0.0/16 のまま適用すると IP pool 待機。
  5. helm install に --create-namespace 忘れ — silent に default に展開。
  6. etcd restore で /var/lib/etcd 直書き — 新規 dir に restore する原則。
  7. upgrade 前に drain せず — workload 中断。
  8. CNI 未インストールで join — node が NotReady のまま。
  9. Kustomize SIG-CLI 公式 docs (kubectl.docs.kubernetes.io) は試験許可外。kubernetes.io/docs 配下のみ。
  10. 静的 Pod を kubectl edit — kubelet が manifest から復元。直接 /etc/kubernetes/manifests/ を編集。
  11. cert 更新後の static pod 未再起動 — apiserver は古い cert を保持。manifest 移動で再起動。
  12. aggregated ClusterRole の rules: [] は正解。"修正" して空でなくすと意図と異なる。
  13. helm template + kubectl apply を "install" 問題でやらない (採点は Helm release を見る)。

D2 よくある落とし穴

  9. cordon は taint node.kubernetes.io/unschedulable を付ける。drain でも cordon と同等の挙動。
  10. maxUnavailable: 0 かつ maxSurge: 0 は無効 (ロールアウト不可)。

NetworkPolicy AND vs OR の覚え方

# OR — ns ラベル "team: foo"  OR  pod ラベル "role: bar"
ingress:
- from:
  - namespaceSelector: { matchLabels: { team: foo } }
  - podSelector:        { matchLabels: { role: bar } }

# AND — "team: foo" の ns 内で、かつ "role: bar" の pod
ingress:
- from:
  - namespaceSelector: { matchLabels: { team: foo } }
    podSelector:        { matchLabels: { role: bar } }   # ← dash 無し！
最大の罠
- podSelector: (dash あり) は OR、 podSelector: (dash なし、namespaceSelector と同じレベル) は AND。採点者は意味で判定するため、YAML の dash 1 つで合否が分かれます。

各タスク開始時のルーチン

  1. タスクを最後まで読む (見落とし防止)
  2. SSH コマンドをコピペで実行
  3. kubectl get nodes で正クラスタ確認
  4. namespace 指定があれば設定
  5. 解く → 検証
  6. exit で SSH 抜ける → 次へ
  7. 7 分超かかったら flag して飛ばす

暗記必須項目

  • vimrc: :set nu sw=2 et ts=2 ai pastetoggle=<F3>
  • etcd 4 flag: --endpoints --cacert --cert --key
  • RBAC: --as=system:serviceaccount:<ns>:<sa>
  • Static pod パス: /etc/kubernetes/manifests/
  • kubeadm upgrade 順序: plan → apply → drain → apt update → restart kubelet → uncordon
  • cert パス: /etc/kubernetes/pki/
  • terminal copy/paste: Ctrl+Shift+C/V
  • busybox:1.28 (nslookup バグなし)
  • kubernetes.io/metadata.name (1.22+ の ns ラベル)
  • HPA は resources.requests 必須

残り 15 分

  • flag 付き問題に再チャレンジ
  • 適用済み manifest の健全性確認 (k get <kind> -A)
  • 早期終了しない — verify を続ける

失敗者がよくやる致命的ミス Top 10

  1. SSH/context 切替忘れ
  2. YAML 手書きしすぎで時間切れ
  3. 低配点問題に >7 分
  4. kubectl explain を使わずドキュメントを彷徨う
  5. vim paste で indent 崩壊 (:set paste 忘れ)
  6. Static pod を kubectl delete
  7. kubectl set image のコンテナ名間違い
  8. verify せず次の問題へ
  9. storageClassName: "" と省略の混同
  10. --record など deprecated 構文に時間を取られる
