Elasticsearch 9.4 vs Qdrant 1.18：ベクトル検索

Elasticsearch 9.4 の DiskBBQ が recall 当たりレイテンシで勝利するのは、その IVF 系のクラスタ配置・逐次読みのディスクレイアウトがクエリコストを recall から切り離すためである。一方 Qdrant の on-disk HNSW + on-disk リスコアは固定のディスク読みコスト床を支払い、その結果 recall 上限が ~0.976 で頭打ちになり、レイテンシが 681 ms 付近に張り付く。

01結論サマリー

本ベンチマークは、根本的に異なる 2 つのディスク常駐型 ANN アーキテクチャを、同一の 21M ベクトル・768-dim の E5 データセット上で、同一の 3x オーバーサンプル + フルプレシジョンリスコアという取り決めの下で対決させる。Elasticsearch 9.4 は DiskBBQ (bbq_disk, bits=2) を用いる。これは階層的 k-means で構築される IVF 系のクラスタインデックスであり、各クラスタはメモリマップファイルから逐次的に読まれる連続した doc-id 順のポスティングリストとして格納され、ネイティブ SIMD でバルクスコアリングされる。Qdrant 1.18 は on-disk HNSW グラフ (m=16, ef_construct=256) を用い、TurboQuant 2-bit コードを RAM に固定 (always_ram=true) する一方、元の float32 ベクトルはディスク上に置き (on_disk=true)、rescore=true がそれら元ベクトルをディスクから再読み込みする。Elasticsearch はあらゆる recall 帯で高速である。~0.89 recall でスループット 1.62x、~0.95 で 2.93x、~0.975 で 5.35x であり、その差は recall とともに単調に拡大する。

決定的な要因は量子化器ではなく on-disk のアクセスパターンである。DiskBBQ は recall ノブ (visit_percentage) を、有界な数の追加的な逐次クラスタ読みへと変換するため、レイテンシは緩やかにしか上昇しない。recall 0.8871 で 106.85 ms、0.9753 で 126.91 ms であり、わずか 147.6 ms で 0.9810 までスケールし続ける。Qdrant の HNSW グラフ巡回は hnsw_ef とともに増大する散在したランダム読み/CPU 律速のホップを発行し、さらに重要なことに、そのリスコアステップはクエリごとに 3x100 のフルプレシジョン 768-dim ベクトルを、ランダム IO のリスコア経路でディスクから引き出す。したがって Qdrant のレイテンシは急峻に上昇し (173.57 -> 337.65 -> 681.42 ms)、その後きつく頭打ちになる。hnsw_ef 100/150/200/256 で recall は ~0.9757 に凍り付き、レイテンシは ~681 ms に凍り付く。この平坦なレイテンシのプラトーこそが動かぬ証拠である。recall が飽和した後は hnsw_ef を追加してもレイテンシが増えない。これはつまり、~681 ms の床が hnsw_ef とは独立した何か、すなわちオーバーサンプル x limit によって決まる固定サイズのディスク律速リスコア読みによって設定されていることを意味する。

誠実な caveat として、これはアーキテクチャ比較であって、単一変数を制御した比較ではない。両システムはインデックス構造 (クラスタリング vs グラフ)、量子化器 (BBQ vs TurboQuant)、そしてリスコア読みが着地する場所 (クラスタ化され直前にスキャンされたブロック vs 別個のランダムアクセスストア) で異なる。いずれも正当な「on-disk・2-bit・cosine・3x リスコア」の構成だが、「比較可能 (comparable)」を「同一 (identical)」と読むべきではない。この結果を最もよく言い表すなら次のとおりである。この規模でのディスク常駐型・高 recall の ANN においては、配置を同じくしたバルクスコアリングを伴う逐次クラスタレイアウトが、リスコアで別個の on-disk float ストアを読みに行くグラフレイアウトに勝つ。

04結果の読み解き

曲線を直接読むと、次のとおりである。

レイテンシの形状。Elasticsearch はほぼ水平である。106.85 ms @ 0.8871 -> 114.75 ms @ 0.9517 -> 126.91 ms @ 0.9753 であり、全スイープは単調で緩やかな成長を示す (108.84, 113.06, 116.87, 120.27, 126.64, 126.91、その後 146.24 @ 0.9784、147.58 @ 0.9810)。0.887 -> 0.981 の recall 全域でコストは ~41 ms である。Qdrant は急峻の後に平坦である。173.57 @ 0.8992 -> 337.65 @ 0.9507 -> 681.42 @ 0.9756、すなわち recall 帯ごとにおよそ 2x のレイテンシ跳躍があり、その後きついプラトーになる。

プラトーが鍵となるデータである。生の CSV では、hnsw_ef = 100, 150, 200, 256 がすべて 681.50 / 681.42 / 681.61 / 681.71 ms で recall 0.9757/0.9756 を返す。recall は小数 4 桁まで平坦、レイテンシは hnsw_ef の 2.5x の変化にわたって 0.3 ms 以内で平坦である。2 つの結論が導かれる。(1) この構成では Qdrant は ~0.976 に recall 上限を持つ。グラフ探索をいくら増やしても、2-bit 一次パス + 300 候補のリスコア予算が決して受け入れない近傍を表出させることはできない。(2) ~681 ms は hnsw_ef から独立したコスト床であり、これはオーバーサンプル x limit = 300 のクエリごとフルプレシジョンベクトルの固定ディスク律速リスコア以外ではあり得ない。対照的に ES はテスト範囲内に上限を持たない。0.9810 に到達し、なお 148 ms 未満を保ちながら登り続けている。

クロスオーバー / recall 制御性。ES の recall は visit_percentage を介して 0.5 刻みで細かく制御可能であり (0.8871, 0.9208, 0.9400, 0.9517, 0.9603, 0.9668, 0.9716, 0.9753, 0.9784, 0.9810)、全域で控えめなレイテンシコストである。Qdrant の hnsw_ef は粗く、かつ飽和的である。実用範囲は事実上 hnsw_ef 10-100 であり、それを超えるとレイテンシで余分なコストは払わないが recall でも何も得られない。

スループット。q/s はレイテンシの物語を映すが、並行性を通じてそれを増幅する。37.02 vs 22.89 (1.62x)、34.55 vs 11.79 (2.93x)、31.24 vs 5.84 (5.35x)。ES は高 recall への登攀全体でわずか ~16% のスループットしか失わない。Qdrant は ~75% を失う。高速化倍率が拡大するのはまさに、ES の平坦なレイテンシがスループットを保つ一方、Qdrant の急峻なレイテンシがそれを破壊するためである。

総括。この数値は 2 つのレジームを描く。ES は各 recall ステップに対して小さく予測可能なレイテンシ増分を支払う (逐次スキャン律速) ため、行儀が良くこの範囲では上限を持たない。Qdrant はエスカレートする巡回コストと固定のリスコア読み床の両方を支払い、それらが相まって recall を ~0.976 で制限し、レイテンシを ~681 ms に張り付かせる。

05なぜElasticsearch 9.4が勝つのか

#1逐次クラスタ読み vs ランダムグラフホップ + ランダムリスコア読み (支配的な原因)

DiskBBQ は IVF 系のクラスタ配置である。階層的 k-means が 21M ベクトルをクラスタにグループ化し、その構成要素は連続した doc-id 順のポスティングリスト (32 件単位のブロック) として格納され、mmap から直接読まれる。visit_percentage を上げると *有界で連続的、プリフェッチに適した* 一組のクラスタ読みが追加されるため、より多くの recall を得るコストはランダム IOPS 律速ではなくほぼ線形でディスク帯域律速である。これが、ES のレイテンシが 0.89 -> 0.95 -> 0.975 の各帯で 106.85 -> 114.75 -> 126.91 ms (0.981 でも 147.6 ms) しか動かない理由である。一方 Qdrant の HNSW は貪欲にノードからノードへホップする。各ホップは、hnsw_ef とともに作業量が増える近傍リストを参照解決し、その後 rescore=true ステップが ~3x100 の元 float32 ベクトル (768-dim、各 ~3 KB) を on-disk の memmap ストアから再読み込みする。Qdrant 自身のドキュメントが、まさにこのパターン、すなわち元ベクトルをディスクに置いたままのリスコアについて『検索速度を著しく低下させ得る』と警告している。これらの組み合わせが、急峻な 173.57 -> 337.65 -> 681.42 ms の上昇を生む。

#2レイテンシが recall から切り離されている (ES) vs レイテンシが結合した後に床に縛られる (Qdrant)

各 visit_percentage の増分が有界な逐次スキャンであるため、ES の recall 対レイテンシ曲線はほぼ平坦であり、recall を生み出し続ける。0.9753@126.91ms、0.9784@146.24ms、0.9810@147.58ms と崖がない。Qdrant はきついプラトーを示す。hnsw_ef=100/150/200/256 はすべて ~681 ms で recall ~0.9757 を返す (生の CSV では 681.50/681.42/681.61/681.71 ms)。このプラトーは機序的に示唆に富む。もし巡回が支配的なら、recall が飽和した後でもレイテンシは hnsw_ef とともに上昇し続けるはずである。hnsw_ef が 2.5x 上がってもレイテンシが ~681 ms に張り付いたままであることは、床が hnsw_ef に対してスケールしないコスト、すなわちオーバーサンプル x limit = 300 のクエリごとフルプレシジョンベクトルの固定サイズ・ディスク律速リスコアによって設定されていることを証明している。その床が recall (~0.976) と速度の両方を同時に制限する。

#3スループット倍率が recall とともに拡大する (1.62x -> 2.93x -> 5.35x)

高速化倍率が一定でないのは、2 つのレイテンシ曲線が乖離するためである。~0.89 recall では ES は 37.02 q/s に対し Qdrant 22.89 q/s (1.62x)。~0.95 では ES が 34.55 q/s を保つ一方 Qdrant は 11.79 q/s に崩落 (2.93x)、~0.975 では ES 31.24 q/s に対し Qdrant 5.84 q/s (5.35x)。ES のスループットは高 recall 帯全体にわたって緩やかに劣化する (37 -> 31 q/s、~16% 低下)。追加される作業がもう数回の逐次スキャンに過ぎないためである。Qdrant のスループットは ~75% 低下する。各 recall の増分が巡回とランダムリスコア読みの量の両方を増大させるためである。差の拡大は、平坦 vs 急峻のレイテンシの直接的な算術的帰結である。

#4量子化器の忠実度は実在するが副次的な寄与要因である

両陣営とも素朴な binary より高精度な 2-bit 方式を走らせているため、ここでは量子化器が主たるレバーでは*ない*。とはいえ量子化器はリスコアに供給される候補の品質を形作る。ES DiskBBQ の bits=2 は非対称エンコーディング (1-2 bit のインデックスベクトル、int4 のクエリ) を用い、さらに符号ビット縮約で失われた大きさ/射影情報を代数的に復元する補正係数を格納し、bits=2 は内部的にオーバーサンプルを 1.5 に自動設定する (ベンチマークではこれに加えてクエリレベルで oversample=3 を適用)。Qdrant の TurboQuant 2-bit (Hadamard 回転 + Lloyd-Max コードブック + 長さの再正規化) は、Qdrant 自身のベンチマークによれば、素の binary 2-bit より 9-24 pp 高精度である。両者とも強力な一次パスのランキングを生み出し、それゆえ両者とも ~0.975-0.98 に到達する。差を分けるのは、巡回とリスコアの最中にディスク上で何が起きるかであって、ビットレベルの距離推定ではない。

06技術ディープダイブ

Elasticsearch DiskBBQ (index_options.type: bbq_disk, bits=2) — IVF 系のクラスタ on-disk 配置
仕組み

DiskBBQ は明示的に Inverted File (IVF) インデックスの発展形であり、HNSW の代替である。グラフでは*ない*。インデックス時、階層的 k-means がベクトルを小さなクラスタ (デフォルト cluster_size 384) に分割し、SOAR バリアントが境界ベクトルを複数クラスタへ溢れさせる。重心が計算され、各クラスタは連続したポスティングリストとして書き込まれる。ポスティングリストは doc ID (昇順 doc-id 順にデルタエンコード、32 件単位のブロック) を格納し、次元は量子化補正と分離されているため、ブロック全体を 1 回の SIMD 演算で補正できる。クエリ時、ES はまず重心をスコアして有望なクラスタを選び、次にその内部の BBQ 量子化ベクトルをバルクスコアリングし、メモリマップファイルからバイトを直接読む。パラメータ意味論に関する注意: visit_percentage は停止前にシャードごとに訪問される候補*ベクトル*の割合 (ベクトル数による停止基準) であって、文字どおりクラスタの割合ではない。クラスタはそのベクトル予算に達するまで重心近接順に訪問される。

この結果への影響

これが ES の平坦なレイテンシ曲線の根本原因である。各 visit_percentage の増分は、バルク SIMD スコアリングされる有界な数の追加的な逐次・プリフェッチ可能なクラスタ読みを加えるため、recall 0.8871 から 0.9810 まで上げるコストはわずか ~41 ms (106.85 -> 147.58 ms) で済む。ランダム IO の爆発もページキャッシュの崖もない。これは、ワーキングセットが RAM を超える 21M ベクトルにおいてディスク常駐型インデックスがまさに必要とするものである。

2 bit での Better Binary Quantization (非対称エンコーディング + 補正係数)
仕組み

BBQ はインデックスベクトルを低ビット深度 (デフォルト 1 bit、ここではより高い忠実度のため bits=2、~16x 圧縮) で、セグメントごと/クラスタごとの重心を中心として格納する一方、クエリはより高い精度 (int4) で保つ。距離はビット単位のカーネルで推定される。int4 のクエリはビットプレーン分解され、AND + popcount + シフト + 和を介してドキュメントのビットと結合され、その後、真の cosine/内積距離を再構成して推定量のバイアスを除去する、格納された per-vector の係数 (重心までの距離、射影 v_c . v_bar) で補正される。bits=2 は内部の oversampling_factor を 1.5 に自動設定する。ベンチマークはその上に明示的なクエリレベルの rescore_vector.oversample=3 を重ねる。再ランキングは巡回から切り離されており (候補収集後に一度だけ行われる)、float32 ベクトルを絶えずページングするのを避ける。

この結果への影響

非対称な int4 クエリ + 補正係数により、一次パスのランキングは十分に正確であり、recall 目標に到達するのにわずかな候補で足りる。スキャンするクラスタが少なく、リスコアも少なくて済む。bits=2 (1-bit のデフォルトに対し) は追加の忠実度を買い、より粗い推定なら頭打ちになりかねない 0.9810 recall まで ES が登り続けるのを助ける。ES 9.x 時点でホットなスコアリング経路はネイティブ C++ SIMD (simdvec カーネル) であり、ブロックスコアリングを ~40% 高速化する。これにより距離計算がクリティカルパスから外れ、ES が逐次レイアウトで最小化してきたディスク I/O だけが支配要因として残る。

Qdrant on-disk HNSW (m=16, ef_construct=256, hnsw_ef をスイープ)
仕組み

HNSW は多層のナビゲート可能なスモールワールドグラフである。検索は疎な最上層から始まって貪欲に下降し、最下層でサイズ hnsw_ef の候補フロンティアを維持する。各ホップはノードの近傍リスト (次数は m=16 で上限) を読み、距離を計算する。この構成ではグラフと 2-bit コードが RAM 上にある (always_ram=true + デフォルトの HNSW 配置) ため、巡回ホップは CPU/RAM 律速である。hnsw_ef を上げるとより多くのノードを訪問し、より多くの距離計算を行う。recall は hnsw_ef とともに上昇するが、近似探索に特有の収穫逓減を伴う。

この結果への影響

hnsw_ef は recall ノブであり、これを増やすと巡回作業が膨らむ。これが、hnsw_ef が 10 -> 50 -> 150 と進むにつれレイテンシが 173.57 -> 337.65 -> 681.42 ms と登る理由の一部である。しかし hnsw_ef 100-256 でのプラトー (recall は ~0.9757 に凍結、レイテンシは ~681 ms に凍結) は、高い側では巡回が拘束的な制約では*ない*ことを示す。リスコアの床こそが制約である。RAM 上のグラフは 1 つのディスクペナルティ (グラフホップのランダム読み) を回避するが、リスコアステップを助けることはできず、Qdrant のディスクコストが実際に着地するのはそこである。

RAM 上の Qdrant TurboQuant 2-bit コード + ディスク上の元 float32 ベクトル (ハイブリッドモード)
仕組み

TurboQuant (quantization_config.turbo, bits2) は回転ベースの PQ 系量子化器である。高速な Hadamard 回転が座標ごとの分散を白色化し、次に各座標が固定の Lloyd-Max コードブック (2-bit では 4 レベル) に対して量子化され、RaBitQ 風の長さ再正規化とセグメントごとの異方性キャリブレーションを伴い、SIMD (pshufb + maddubs/VPDPBUSD) でスコアされる。always_ram=true は高速な一次パススコアリングのためコンパクトな 16x 圧縮コードを RAM にキャッシュする。on_disk=true は元の float32 ベクトルを別個のメモリマップ on-disk ストアに保つ。これは Qdrant が文書化しているハイブリッドの「元ベクトルはディスク、量子化は RAM」モードである。(注: リポジトリの README はこれを『binary quantization (encoding: two_bits)』と誤記しているが、実際のスキーマは TurboQuant を用いており、これは同じ 2-bit/16x ストレージ階級にある別個の機能である。)

この結果への影響

TurboQuant は Qdrant に強力な一次パスランキングを与える (~0.9757 に到達する) ため、量子化器は弱点ではない。弱点は構造的である。float32 の元ベクトルは*別個の* on-disk ストアに存在するため、oversampling=3 を伴う rescore=true は、クエリごとに ~300 のフルプレシジョン 768-dim ベクトルをランダム読みでリスコア経路から取得しなければならない。これが Qdrant の曲線を制限する固定の ~681 ms の床であり、まさに Qdrant のドキュメントが元ベクトルがディスク上にあるときの主要なスローダウンとして指摘するパターンである。

二段階のオーバーサンプル + フルプレシジョンリスコア (共有の取り決め、非対称な IO コスト)
仕組み

両エンジンとも、フェーズ 1 を圧縮コード上で走らせて oversample x k の候補を集め、次にフェーズ 2 で生存者を元の float32 ベクトルに対して再ランキングする。ES の rescore_vector.oversample=3 と Qdrant の oversampling=3 は、いずれも top-100 のクエリに対して 300 候補を事前選択する。決定的な違いは、float32 の読みが*どこに*着地するかである。DiskBBQ ではリスコア候補は直前にスキャンされた同一の連続クラスタブロックから来て、mmap から逐次的に読まれる。Qdrant ではそれらは別個の on-disk float ストアに散在しており、新たなランダム読みを要する。

この結果への影響

オーバーサンプル係数は同一であるため、それは差を説明しない。しかしその IO フットプリントは差を説明する。DiskBBQ のリスコアは既に実行した逐次スキャンに償却される。Qdrant のリスコアはクエリごとに ~300 のランダムディスク読みの新たなバッチであり、オーバーサンプル x limit によって固定され、hnsw_ef とは独立である。これこそが、recall が飽和しても Qdrant のレイテンシ床 (~681 ms) が下がらない理由である。リスコア読みのコストは hnsw_ef をどうチューニングしても一定なのである。

両クラスタ下の Hyperdisk-balanced ボリューム (160,000 IOPS / 2,400 MB/s)
仕組み

各データ Pod は 160k IOPS、2,400 MB/s 定格の 200Gi GCP hyperdisk-balanced ボリュームを持つ。これは両アーキテクチャのディスクアクセスが供給される物理基盤である。逐次バルク読み (DiskBBQ) は帯域を効率的に消費する。小さなランダム読み (溢れた場合の Qdrant のグラフホップ、およびリスコアの float 取得) は IOPS とラウンドトリップを消費する。

この結果への影響

160k IOPS の高速ディスク上であっても、アクセス*パターン*が支配する。クラスタレイアウトは recall を帯域律速の逐次読み (安い) へと変える一方、別個 float ストアのリスコアは recall をリクエストごとのラウンドトリップレイテンシを伴うランダム IOPS 律速の読み (高い) へと変える。したがって同一ハードウェアが ES では平坦なレイテンシを、Qdrant では急峻なレイテンシをもたらす。これは、決め手がディスク速度ではなくアーキテクチャであることのきれいな実証である。

07留意点・公平性

  • 単一変数の比較ではない。両陣営は同時に 3 つの軸で異なる。すなわちインデックス構造 (ES のクラスタ/IVF DiskBBQ vs Qdrant の HNSW グラフ)、量子化器 (BBQ vs TurboQuant)、リスコア IO の局所性 (配置を同じくした逐次ブロック vs 別個のランダムアクセス float ストア) である。見出しの結果はアーキテクチャ比較であって、いずれか一要因の分離ではない。リスコアストアの差が作業の多くを担っていることを認めずに、勝利を純粋に『DiskBBQ』に帰することはできない。
  • 『HNSW m=16』は共有のノブではない。リポジトリの構成表は ES について『-』を記載している。DiskBBQ は HNSW を一切用いない (m パラメータがなく、visit_percentage でチューニングする) ためである。m=16 を比較可能な設定として提示するのは誤解を招く。両エンジンは根本的に異なるインデックスファミリを用いている。
  • リポジトリ README の量子化器ラベルは不正確である。README の本文は Qdrant 側を『binary quantization (encoding: two_bits)』と呼ぶが、実際のスキーマは TurboQuant (quantization_config.turbo, bits2) を用いている。これは回転ベースの PQ 系の手法であり、Qdrant が binary 2-bit とは別個 (かつ 9-24 pp 高精度) であると文書化しているものである。両者とも 2-bit/16x 階級にあるため『比較可能なビット深度』は成立するが、『同じ量子化手法』は成立しない。
  • visit_percentage の意味論はよく誤って述べられる。これは停止基準としてシャードごとに訪問される候補*ベクトル*の割合であって、nprobe 風のノブとしてスキャンされるクラスタの割合ではない。ES の recall 制御は実在するが、その機序は正確に記述されるべきである。
  • 非対称なストレージモード。Qdrant はハイブリッドで動作する (always_ram=true により 2-bit コードを RAM、元ベクトルをディスク) ため、そのグラフホップは RAM 律速であり、ディスク律速なのはリスコアのみである。ES DiskBBQ は量子化ベクトルをディスクに保ちストリームする。したがって『on-disk』は両陣営で異なる意味を持つ。Qdrant にとって決定的なディスクコストは、グラフ巡回ではなく具体的にリスコアステップである。
  • リスコアのチューニングはスイープされていない。Qdrant の ~681 ms の床は oversampling=3 x limit=100 = 300 のディスク読みによって駆動される。oversampling を下げる、リスコアを無効化する、あるいは inline_storage を有効化する (ベクトルを配置同居させてリスコア IO を削減するため 3-4x のストレージ) ことで、Qdrant の数値は実質的に変わり得る。これらの緩和策はいずれも行使されていない。この比較は選択された 3x リスコア・ディスク読みの構成を反映しており、それは Qdrant 自身のドキュメントが既知の遅い経路として指摘するものである。
  • レプリケーションの非対称性。ES はレプリカ 1、Qdrant は replication_factor 2 を用いる。3 シャードの場合、これはノードごとのデータ量、ページキャッシュ圧、そしてディスクのワーキングセットがどう分散されるかを変え得る。これは、ここで支配的なディスク IO 律速レジームにとって実在するが未定量の交絡である。
  • シャーディング/セグメントの影響は考慮されていない。両者とも 3 シャードを用いる。ES はシャード内のすべての Lucene セグメントにクエリし、評価時のマージ状態がレイテンシに影響する。ベンチマークは await-index-ready が ES のマージと Qdrant の最適化を待つことに言及しているが、セグメント/クラスタ数の残存差は制御されていない。
  • recall 上限は構成固有であり絶対的ではない。Qdrant の ~0.976 のプラトーは、この量子化器 + 300 候補のリスコア予算の性質であって、HNSW 固有の限界ではない。より大きなオーバーサンプルや異なる量子化はそれを引き上げ得る。逆に、DiskBBQ は非常に高い recall (99%+) では多くのクラスタを訪問せねばならず遅くなると文書化されている。テストされた範囲 (<=0.981) は DiskBBQ のスイートスポットに収まっている。
  • ネイティブコードの対等性。両エンジンとも今やホットな距離/スコアリングカーネルをネイティブ SIMD で走らせる (ES は Panama FFI/simdvec を介して、Qdrant は独自の SIMD カーネルを介して) ため、この差は『JVM vs ネイティブ』の計算経路の話ではない。それは真にディスクアクセスパターンに関するものである。

08結論

Elasticsearch 9.4 DiskBBQ が本ベンチマークで勝利する。スループットは 1.62x から 5.35x で recall とともに拡大する。それは、より良い量子化器のためではなく、そのインデックスがどうディスクに触れるかのためである。DiskBBQ の IVF 系クラスタレイアウトは各 recall 増分を有界で連続的、SIMD バルクスコアリングされた逐次読みへと変えるため、レイテンシはほぼ平坦に保たれ (106.85 -> 147.58 ms)、recall は崖なく 0.98 を超えて登り続ける。Qdrant 1.18 の on-disk HNSW は recall に対して二重に支払う。hnsw_ef の上昇が巡回を膨らませ、oversampling=3 を伴う rescore=true がクエリごとに ~300 のフルプレシジョン 768-dim ベクトルを別個の on-disk ストアからランダム読みで取得する。そのリスコアは hnsw_ef から独立した固定コストであり、これが、Qdrant のレイテンシが ~681 ms に張り付き、その recall が hnsw_ef 100-256 で ~0.976 に頭打ちになる理由である。このプラトーはディスク律速リスコア床の直接的な署名である。より深い教訓はこれら 2 製品を超えて一般化する。数千万ベクトル規模のディスク常駐型・高 recall ANN において、配置を同じくしたバルクスコアリングを伴う逐次クラスタレイアウトはクエリコストを recall から切り離す一方、リスコアで別個のランダムアクセス float ストアを読むグラフレイアウトはそれらを結合し、壁にぶつかる。この結果はリポジトリの CSV から実在し再現可能だが、それはアーキテクチャレベルの判定である。異なる構造 (クラスタリング vs グラフ)、異なる量子化器 (BBQ vs TurboQuant)、異なるリスコア IO の局所性がすべて連動して動く。ここでの Qdrant の数値は、自身のドキュメントが遅いと指摘する 3x リスコア・ディスク読みの経路を反映している。緩和策 (オーバーサンプル低減、リスコア無効化、inline_storage) はテストされておらず、差を縮め得る。実行された構成の範囲内では、Elasticsearch DiskBBQ が recall 当たりレイテンシで明白な勝者であり、このディスク常駐型・高 recall レジームにとってアーキテクチャ的により健全な選択である。

この分析の作り方: 数値はすべて本リポジトリの実測結果 CSV に基づきます。「なぜ」の技術解説は、複数エージェントによる一次資料（Elastic search-labs / Lucene / OpenSearch / Qdrant の公式ドキュメント・ブログ）の並列 Web 調査と、主要主張に対する敵対的事実検証（反証を試みる立場でのファクトチェック）を経て統合したものです。検証で訂正された点（例: DiskBBQ は HNSW ではなく IVF 系クラスタ構造、Qdrant 側は TurboQuant であり README の「binary quantization」表記は不正確、など）は本文・留意点に反映済みです。
