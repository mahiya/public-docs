Vendor View — 01

Anthropic の Context Engineering

「有限のアテンション予算をどう使い切るか」。Anthropic はコンテキストを枯渇する資源として捉え、最小で最高シグナルのトークン集合を見つけることを設計の中心原理に据える。理論・原典寄りの視点。

出典: effective-context-engineering-for-ai-agents.md（Engineering at Anthropic, 2025-09-29）／ tool-use-context-engineering-context-engineering-tools.md（Claude Developer Platform Cookbook）

01定義

Anthropic は context engineering をプロンプトエンジニアリングの自然な発展形と位置づける。プロンプトが「最適な言葉の選び方」だったのに対し、context engineering は「どんなコンテキストの構成が望ましい挙動を生む可能性が最も高いか」という、より広い問いに答える営みである。

Definition

Context engineering とは、LLM の推論中に最適なトークン（情報）の集合をキュレーションし維持するための戦略群である。「コンテキスト」とは LLM からサンプリングする際に含まれる全トークンを指し、システム指示・ツール・MCP・外部データ・メッセージ履歴など、プロンプト以外に紛れ込むすべてを含む。

プロンプト作成が一度きりの離散的タスクであるのに対し、context engineering は反復的であり、「モデルに何を渡すか」を決めるキュレーション工程が推論のたびに発生する。ループで動くエージェントは絶えず新たな情報を生み出すため、その膨大な候補から限られた窓に入れるものを選び続ける必要がある。

02なぜ重要か — 有限なアテンション予算

LLM は人間と同様、ある点を超えると焦点を失う。トークン数が増えるほど正確な想起が困難になる現象 — コンテキストロット（context rot） — がすべてのモデルで観測される。

  • アテンション予算：Transformer は全トークンが互いに注意を向け合うため、n トークンで n² の関係が生じる。コンテキストサイズとアテンションの焦点はトレードオフ。
  • 訓練分布の偏り：短い系列のほうが学習データに多く、長距離依存への経験とパラメータが相対的に乏しい。
  • 結果として「崖」ではなく性能の勾配（gradient）が生じる。長文でも有能だが検索精度と長距離推論は落ちる。
良いコンテキストエンジニアリングとは、望ましい結果の確率を最大化する「最小で最高シグナルなトークン集合」を見つけること。

03効果的なコンテキストの解剖学

◎システムプロンプト

「ちょうど良い高度（right altitude）」で書く。脆い if-else と曖昧な一般論の間。XML タグや Markdown 見出しでセクション化し、挙動を完全に示す最小限の情報を狙う（最小＝短いとは限らない）。

⚙ツール

エージェントと情報・行動空間の契約。トークン効率の良い結果を返す。機能の重複がなく自己完結し用途が明確に。肥大したツール群は最大の失敗モード。

✎例（few-shot）

エッジケースを羅列せず、期待挙動を的確に描く多様で正典的な例を厳選。例は「千の言葉に値する絵」。

↻メッセージ履歴

全体を「情報豊かでありながら引き締まった（informative, yet tight）」状態に保つ。

04コンテキストの取得 — Just-in-time とエージェント検索

  • 軽量な識別子を保持：ファイルパス・保存クエリ・Web リンク等の参照だけ持ち、実行時にツールで動的ロード。Claude Code は head/tail・glob/grep で全ロードせず分析。
  • メタデータが挙動を導く：フォルダ階層・命名規約・タイムスタンプがシグナルに。
  • 漸進的開示：探索で段階的にコンテキストを発見し、作業記憶には必要なものだけ。
  • ハイブリッド戦略：速度のため一部前取得しつつ裁量で自律探索。CLAUDE.md 前ロード＋JIT。法務・金融など動的でない領域は事前取得寄り。
"do the simplest thing that works"。モデルが賢くなるほど人手のキュレーションは減る。

05長期ホライズンタスクの技法

⊟Compaction

上限に近づいた会話を要約し新窓を再起動。決定・未解決バグ・実装詳細を保ち冗長なツール出力を捨てる。まず recall 最大化→precision 向上。

▤Structured note-taking

窓外の永続ストレージにメモを書き後で引き戻す。NOTES.md・TODO。Pokémon プレイ例で数千ステップの目標・地図・戦略を維持。

⫶Sub-agent architectures

クリーンな窓を持つ専門サブエージェント。数万トークン探索しても返すのは1,000〜2,000トークンの蒸留サマリのみ。複雑リサーチで単一を上回る。

063つのプリミティブ（Cookbook の実装視点）

プリミティブ作用対象失うもの解決する問題
Compaction現窓の会話全体逐語の詳細（細部は欠落しうる）セッション内の全成長
Tool-result clearing現窓のツール結果古いツール結果（必要なら再取得）ツール結果の肥大
Memory窓外の外部ストレージ呼び出しオーバーヘッド／保存判断の質次第セッション横断の永続化
Key takeaway

問うべきは「3つ全部使うべきか？」ではなく「自分のワークロードが3つの問題のうちどれを実際に抱えているか？」。ロス性はバイナリではなくスペクトラム。

Vendor View — 02

OpenAI の Context Engineering

理論よりも実装パターン。Agents SDK の Session を使い、短期記憶（trimming / summarization）と長期記憶（state ベースのパーソナライズ）をどう作り、どう評価するかを具体的なコードで示す。合言葉は「evals is all you need」。

出典: session_memory.md（Short-Term Memory Management with Sessions）／ context_personalization.md（Long-Term Memory Notes）（developers.openai.com cookbook）

01定義

Definition

Context engineering の核心は「ある瞬間にモデルが何を知っているかを形作ること」。何を保存し（store）、何を呼び戻し（recall）、何を作業記憶に注入する（inject）かを管理し、パーソナルで一貫性ある文脈理解エージェントを作る。

「コンテキスト」はモデルが一度に注意できる入出力トークンの窓（GPT-5 で入力 272k／出力 128k）。大きな窓でも未整理の履歴・冗長なツール結果・ノイズの多い検索で溢れる。コンテキスト管理は最適化ではなく必須事項。「応答する」から「記憶する」への飛躍が新フロンティア。

02なぜ重要か

  • 長スレッドでの一貫性維持：「昨日の計画」が「今日の依頼」を上書きしない。
  • ツール呼び出し精度の向上：焦点の合った文脈が関数選択・引数充填を改善。
  • レイテンシ・コストの低減。
  • エラー／ハルシネーション封じ込め：要約は「クリーンルーム」、trimming は context poisoning を回避。
  • デバッグ・可観測性：安定サマリと境界明確な履歴でログを比較可能に。

03短期記憶：Trimming と Summarization

✂Context Trimming

直近 N ターンだけ保持（TrimmingSession, max_turns）。決定論的・追加レイテンシゼロ・逐語保持だが、長距離の制約や ID は N を過ぎると突然消え「健忘」に見える。

⊟Context Summarization

直近 N ターンは逐語で残し古い履歴を構造化サマリに圧縮（SummarizingSession）。長距離をコンパクトに保つが、要約ロス・遅延・poisoning リスク。

観点TrimmingSummarizing
レイテンシ／コスト最小サマリ更新時に上昇
長距離の想起弱い（ハードカットオフ）強い（繰り越し）
リスク種別コンテキスト喪失歪み／poisoning
向く用途ツール多用・短いワークフローアナリスト／長スレッド

サマリプロンプト設計：マイルストーン／用途特化／矛盾チェック／タイムスタンプと時系列／チャンク化／ツール性能の知見／例示／ハルシネーション抑制（不確実は UNVERIFIED）。「次の担当者への引き継ぎ」の感覚で書く。

04長期記憶：State ベースのパーソナライズ

▦構造化メモリ

スキーマ駆動・機械強制可能。顧客ID・パスポート期限・ロイヤルティ・座席選好・保険プロファイル等。ロジック／予約 API に直接使える。

≈非構造化メモリ

ナラティブ・意味的。「通路側を好む」等。推論・パーソナライズに最適化。

スコープ分離：耐久的選好は Global、今回限りは Session（staging）。設計指針は「同じ仕事を人間がやるなら作業記憶に何を保持するか？」。

Inject 注入→ Reason 推論→ Distill 蒸留→ Consolidate 統合
  • Injection：構造化を YAML フロントマター、非構造化を Markdown で system prompt へ。<memory_policy> で扱いを指示。
  • Distillation：会話中に save_memory_note で候補メモ捕捉（memory-as-a-tool）。耐久的・行動可能・明示的なものだけ。
  • Consolidation：終了時に session→global 昇格。最もデリケート。重複排除・矛盾解消（recency wins）・忘却を明示的に。忘却は必須。
優先順位ルール
  1. 現在の対話のユーザー最新指示が最優先。
  2. 構造化プロファイルキーは信頼。
  3. Global メモリは助言的で現在の指示を上書きしない。
  4. Session メモリは今回限り Global を上書き。
  5. 衝突時は明確化の質問を1つだけ。

05セキュリティ：記憶は高価値の攻撃面

脅威は context poisoning・instruction injection・over-influence。

段階ガードレール
Distillation機微パターン拒否、指示型ペイロード拒否、ツールスキーマを承認フィールドに限定
Consolidationno invention、recency wins、意味的重複排除、TTL 減衰
Injection明示デリミタ、優先順位（現在の発話 > session > memory）、recency 重み付け、advisory 扱い
記憶が挙動を変えうるなら、capture・consolidation・injection の各時点で安全チェックを通過しなければならない。

06評価 — evals is all you need

メモリは強い時間依存を持ち、オーケストレーションパイプラインなので distillation・consolidation・injection をエンドツーエンドで評価。指標：memory_write_rate・blocked_write_rate・memory_conflict_rate・time_to_personalization。

Philosophy

ゼロショットから始め、eval ハーネスで実失敗を集め、必要なら小さな「memory specialist」へ fine-tune。記憶システムは計測された反復で良くなる。

Google の Context Engineering

Google は context engineering を「AI にとっての意味のアーキテクチャ」と表現し、Google Cloud のインフラ（Vertex AI / BigQuery / Agent Platform）と Gemini Embedding に紐付けて語る。インフラ・基盤寄りの視点。

01定義

Definition

Context engineering は「AI にとっての意味のアーキテクチャ」。単発の付箋を渡すのではなく、ラベル付きキャビネット（BigQuery）・ライブ接続（Agent Platform）・明確なルールを備えた「ハイテクな作業場」を用意し、AI が当て推量せずデータ駆動の現実で動くようにする。

観点レガシーなプロンプトモダンな context engineering

スケーラビリティ再現困難Google Cloud に内蔵

02コンテキストの3層モデル

▣Persistent
永続層

システム指示。AI の世界の「物理法則」。役割・トーン・許可／禁止。全インタラクションで有効。

◐Semi-persistent
半永続層

メモリ。会話履歴とユーザー選好を追跡し、繰り返させない。

⚡Transient
一時層

動的データ。Agent Search の文書・ライブ API・思考用メモ。リクエスト毎に変わる。

032M トークン経済と Context Caching

1M–2M
Gemini 3.1 のコンテキスト窓（トークン）
90%
Context Caching によるトークンコスト削減

トークンは LLM の「RAM」。巨大窓でコードベース全体・1時間動画・数千行の BigQuery を一度に渡せる。Context Caching は背景データを 90% 割引でアクティブメモリに保持し、都度送信を不要にする。

巨大窓は「埋めれば性能が上がる」わけではないとする Anthropic / Elastic とは対照的に、Google はインフラ（巨大窓＋キャッシュ）で枠を広げて経済性を変える方向を前面に出す。

Elastic の Context Engineering

Elastic は context engineering を「適切な情報を適切なタイミングで AI に与える実践」と定義し、その全構成要素を 1つのプラットフォーム（Elasticsearch ＋ Agent Builder）に集約できると主張する。製品・実装側の視点。

出典: elastic/01〜04（context-engineering-overview / vs-prompt-engineering / what-is-context-engineering / context-engineering-for-ai-agents）（elastic.co Search Labs ほか）

01定義

Definition

Context engineering は「適切な情報を適切なタイミングで与える実践」。新しい同僚へのブリーフィングのように、全社文書を積むのではなく具体的タスクに最も関連する情報を選んで渡す。モデルを取り巻く情報エコシステム全体を設計する。

ソフトウェア工学の原則から着想。DB・API・パイプラインを設計するように、知的エージェントを支える情報アーキテクチャを設計する。プロンプトエンジニアリングはその部分集合。

観点プロンプトエンジニアリングコンテキストエンジニアリング
中心の問い「どう言い回すか？」「モデルは何を知る必要があるか？」
スコープ単一クエリシステム全体の情報フロー
失敗モード曖昧さ（誤解釈）検索の問題（誤文書・陳腐化・溢れ）
デバッグ手法言語的精度データアーキテクチャ（検索調整・剪定・順序）

02コンテキストの構成要素（5〜6要素）

▷User prompt

directive・exemplars・output formatting・style・role・additional information から成る。

⌕RAG

▤Memory

短期（state/history）と長期（セッション横断）。意味検索で過去メモリを引き戻す。

⚙Tools

description・型付きparameters・実行関数・（任意）output schema。

{}Structured outputs

JSON 等のスキーマ準拠出力。下流との「契約」。

03検索の作り込み — Elastic の本領

「窓が広がったから RAG は死んだ」に対し、余剰情報がもたらす context confusion を理由に反論。

⧉Semantic chunking

固定長で切らず段落・関数・論理セクション単位でまとめ、周辺も含める。

⚖Hybrid search

ベクトル（概念類似）＋ BM25（厳密一致 SKU-123AB）を1クエリで統合。標準提供。

↑Reranking

上位100件を高速取得→高精度リランカーで再スコア。lost-in-the-middle を緩和。

04ツールのオーケストレーションと選択

100ツールのエージェントは窓の 30〜40% を説明に費やしうる。簡潔・曖昧さなし・合成可能・名前空間で選択的ロード・tool result filtering（要点のみ）。

3戦略：Agentic search（自然言語→ES|QL、要約だけ返すサブエージェント）／Domain workflows（決定論的チェーン、state machine/DAG）／Dynamic tool discovery (RAG-MCP)：ツールメタデータをベクトルDBに格納し上位 N だけ取得。呼び出し精度3倍・トークン50%超削減。

05パイプラインと4戦略

① 取得・生成→ ② 処理（最適化）→ ③ 管理（短期／長期）
戦略内容
Selection（選択）窓外に置き必要時に JIT 取得。ハイブリッド（キーワード＋意味＋グラフ）で絞る。
Writing（書き込み）外部メモリ（scratchpad/DB）にオフロード。計画を書き出す。
Compression（圧縮）Summarization と Trimming（古いメッセージ・冗長ツール出力を削除）。
Isolation（分離）独立窓のサブエージェントに sub-task。リードは蒸留結果のみ受領。

06中核の制約と失敗モード

制約：context rot・$O(n^2)$・lost in the middle・ノイズ。本番の失敗モード：context poisoning（OWASP LLM Top 10）・goal drift・capacity overflow。

07Elastic の優位性 — 1プラットフォームに集約

Elasticsearch はベクトルDB・検索エンジン・NoSQL 文書ストアを統合し、context engineering の要素を単一システムで満たせる。

▥1つのデータストア

チケット・ログ・文書等の非構造化データをカラムナ保存で統合しリアルタイム検索。

⚖最良の関連性エンジン

ハイブリッド検索・セマンティックリランキング・GPU 内蔵推論（Jina AI／第三者）。

⚒Agent Builder

ES|QL カスタムツール・データチャット・外部エージェント統合。数分で出荷。

🛡安全に、すべて計測

文書・フィールド級 RBAC、テレメトリ、ガードレール、オンプレ／エアギャップ、検索評価（nDCG）。

Synthesis — Elastic Perspective

統合まとめ：1つの定義・要素・設計ポイント

4社の主張を1つに束ね、統一定義／要素／各要素の詳細な設計ポイントとして整理する。最後の景色は、顧客が Foundry + Azure AI Search で AI エージェントを開発する中へ Elastic を提案する立場から描く。

011つの定義

Anthropic「最適トークン集合のキュレーション」、OpenAI「ある瞬間にモデルが何を知っているかを形作る」、Google「意味のアーキテクチャ」、Elastic「適切な情報を適切な時に与える実践」。これらを束ねると —

Unified Definition

Context engineering とは、LLM／エージェントが望ましい挙動を一貫して生み出すために、推論のたびに限られたコンテキストウィンドウ（作業記憶）へ入れる情報を、能動的に選び・構造化し・管理するシステム設計の規律である。

プロンプトエンジニアリングが「どう言い回すか（単一ターン）」を扱うのに対し、context engineering は「モデルは何を知る必要があるか（システム全体の情報フロー）」を扱い、プロンプトエンジニアリングはその部分集合。EA共通の指導原理は「望ましい結果の確率を最大化する、最小で最高シグナルなトークン集合を見つけること」。

024社が共有する前提

共通認識内容
コンテキストは有限資源AGE が context rot・$O(n^2)$・lost in the middle を指摘。窓は埋めれば良いわけではない。
JIT（必要時取得）への移行A の just-in-time、E の Selection、G の Transient 層。
記憶のロス性とガバナンス誤りは context poisoning として波及。O はガードレール＋evals、E は RBAC。
ツールは最小限で曖昧さなくA「人間が選べないなら AI も選べない」、E の RAG-MCP。
モデル向上＝人手キュレーション減A。ただし巨大窓でも working set を絞る価値は残る。

強調点の違い：A理論と長期ホライズン技法／OSDK 実装と evals／Gインフラ（巨大窓・キャッシュ・embedding）／E検索プラットフォームへの集約。視点は補完的。

04各要素の詳細な設計ポイント

1
システム指示
役割・規則・ガードレールを定める永続層
設計ポイント
  • A「ちょうど良い高度」で書く。脆い if-else と曖昧な一般論の間。挙動を完全に示す最小限の情報（最小＝短いではない）。
  • AEXML タグ／Markdown 見出しで区切り、抽象ルールより具体例を。
  • E特異性・柔軟性・簡潔性を両立。位置バイアスを踏まえ重要な制約は目立つ位置へ。蓄積で押し出されると behavioral drift。
  • G「AI の世界の物理法則」。全インタラクションで有効に。
  • Aまず最良モデル＋最小プロンプトで試し、失敗モードに応じ指示・例を足す。
2
ユーザープロンプト
挙動を起動する最も注意重みの高い入力
設計ポイント
  • E6要素：directive・exemplars・output formatting・style・role・additional information。
  • E最高の注意重みを持ち他所の矛盾を上書き。session 固有情報の注入口に。
  • E最も予測不能。query understanding 再定式化・prompt injection 検知・フォールバックで吸収。
  • E算術・推論には zero-shot より chain-of-thought。実例：曖昧な「良い本を推薦」は失敗、ジャンルと具体例で 103,063 件から的確に検索。
3
検索（RAG / リトリーバル）
外部知識を JIT で取得しグラウンディング
設計ポイント
  • ERAG は3限界（ドメイン知識欠如・新情報・ハルシネーション）に対処。context confusion を理由に「RAG は死んだ」に反論。
  • ESemantic chunking／Hybrid search（ベクトル＋BM25）／Reranking（上位100→高精度再スコア）。
  • AJIT＋ハイブリッド戦略：軽量参照を保持し実行時ロード。速度 vs 精度。動的でない領域は事前取得寄り。
  • GEmbedding が中核：文書・履歴・ツール定義を作業記憶へ統合。Matryoshka で次元圧縮。
4
メモリ（短期・長期）
セッション内の継続性と横断の永続化
短期記憶
  • OETrimming（直近 N ターン、決定論的・逐語だが長距離は突然消失）とSummarization（圧縮、長距離保持だがロス・遅延・poisoning）。
  • Oサマリは「引き継ぎ」の感覚で。マイルストーン・時系列・チャンク化・不確実は UNVERIFIED。
長期記憶
  • Ostate ベース：構造化プロファイル＋非構造化ノート。Global/Session 分離。ライフサイクル Inject→Reason→Distill→Consolidate、忘却は必須。
  • Anote-taking / memory tool：窓外ストレージにメモ。check-memory-first。CLAUDE.md（ユーザー）＋auto memory（モデル）。
  • E語彙・意味・ハイブリッド検索で過去メモリ取得。優先順位は現在の発話 > session > global（助言的）。
5
ツール
外部システムと対話し追加コンテキストを引き込む契約
設計ポイント
  • AE最小限・曖昧さなし・1機能1ツール。tool result filtering で要点のみ返す。
  • Eツール定義は窓の 30〜40% を食いうる。合成可能な小ツール、名前空間で選択的ロード。
  • EAgentic search／Domain workflows／RAG-MCP（呼び出し精度3倍・トークン50%超削減）。
  • Aself-contained・error 頑健・記述的パラメータ。Gツール定義自体を embedding して選択。
6
構造化出力
下流システムとの契約となるスキーマ準拠の出力
設計ポイント
  • EJSON/XML/テーブル等。REST の OpenAPI 契約に相当。function calling と組み合わせ標準契約を定める。
  • E構文不正な JSON に備え優雅なエラー処理。出力形式の明示で parsing エラー減・品質向上。
  • O記憶サマリも見出し付き構造化フォーマットにすると後続処理・評価が容易。

06Elastic 視点での実装 — 1プラットフォームに集約

Elastic への対応づけ

顧客は Foundry + Azure AI Search でエージェント構築中。Elastic を重ねる価値は「検索（要素3）の関連性とメモリ（要素4）の永続化を1ストアで、ハイブリッド＋リランクで高精度に、RBAC・エアギャップ込みで」提供できる点。

統合要素Elastic の手段
3. 検索（RAG）BM25＋kNN のハイブリッド検索標準、セマンティックリランキング、EIS の GPU 内蔵推論（Jina AI／第三者）。意図解釈・権限フィルタ・ランキング。
4. メモリ長期メモリストアとして語彙・意味・ハイブリッド検索で過去セッション取得。多様な非構造化データを1ストアに。
5. ツールAgent Builder で ES|QL ツール登録。メタデータを索引し RAG-MCP で動的選択。
1・2・6.Agent Builder がプロンプト・データ・ワークフローを接続。構造化／非構造化を両立。
窓管理・ガバナンスagentic search による分離、RBAC（文書・フィールド級）、テレメトリ、nDCG 評価、オンプレ／完全エアギャップ。

相対性ジャーニー：関連性から始める→コンテキストを設計（Jina＋リランカー）→エージェントをオーケストレーション（Agent Builder）。オープンな Inference API と LangChain・LlamaIndex・MCP 統合で自前モデルも持ち込める。

結論

4社の context engineering は視点（理論／実装／インフラ／プラットフォーム）が違うだけで同じ規律を語る。共通原理は「有限の窓へ、最小で最高シグナルな情報を、能動的に選び・構造化し・管理する」。6要素のうち関連性の高い検索とスケールするメモリが最も差がつくレバーで、Elasticsearch ＋ Agent Builder はそこを1プラットフォームで埋める提案足りうる。
