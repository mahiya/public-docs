Elasticsearch 9.3 vs OpenSearch 3.5：ベクトル検索

Elasticsearch 9.3 の BBQ HNSW が両軸で勝利する。その理由は、補正係数を伴う非対称量子化（クエリ int4 / ドキュメント 1-bit）が、OpenSearch の FAISS 32x 対称 1-bit BQ よりも正確なランキング信号をもたらすためである。これにより ES はより高い recall 対 candidates 曲線上に位置し、さらにそのビット演算 SIMD 距離カーネルは candidate あたりのコストが格段に安いため、num_candidates の増加に伴って拡大する 5.6x〜9.0x のサーバレイテンシ低減を実現する。

01結論サマリー

このベンチマークは、アーキテクチャ的に類似した 2 つのシステムを対決させる。両者とも、128 次元ベクトルを 1 次元あたり 1-bit に量子化したものの上に HNSW グラフ（m=16, ef_construction=100）を構築し、フルプレシジョンの float でリスコアする。これを同一構成の 6 ノード GKE クラスタ（e2-standard-16、3 シャード / 1 レプリカ）上で実行する。Elasticsearch 9.3 は Better Binary Quantization（bbq_hnsw）を使用し、OpenSearch 3.5 は FAISS エンジンを compression_level=32x（1-bit バイナリ量子化）の in_memory モードで使用する。スイープパラメータは num_candidates（250..10000）であり、これが両者の k と HNSW のビーム幅の両方を駆動する。検証済みの結果は明白である。マッチさせた全ての num_candidates において、ES は recall@100 をより高く達成し、しかもサーバレイテンシは劇的に低い。nc=250 で 8.01 ms 対 45.07 ms（5.6x）、nc=1000 で 13.56 ms 対 62.93 ms（4.6x）、nc=5000 で 50.58 ms 対 252.93 ms（5.0x）、nc=10000 で 90.34 ms 対 810.91 ms（9.0x）である。

これを生み出すメカニズムは 2 つあり、それぞれ曲線の異なる部分に作用する。第一に、ランキング品質の効果が低域を支配する。ES の recall リードは nc=250 で 6.8 パーセントポイント（0.7704 対 0.7023）だが、nc=10000 では 0.08 pp（0.9848 対 0.9840）まで縮小する。このパリティへの収束は、より優れた量子化距離の推定（ESTIMATE）であることの指紋である。BBQ はクエリを int4（4 ビット/次元）に保ちつつドキュメントを 1 bit とし、さらに格納された補正係数を加えることで、符号ビットが潰してしまう大きさ（magnitude）情報の大部分を代数的に復元する。一方、OpenSearch 3.5 のデフォルトの対称パスは両側を 1 ビットに量子化するためクエリの大きさ情報を失い、同点（tie）の多い、歪んだランキングを生み出す。より正確な推定はより小さなプールの中で真の近傍を見つけるため、ES はより低い num_candidates で任意の目標 recall に到達する。プールが十分に大きくなると、両エンジンとも最終的に同じ近傍を浮上させ、recall は収束する。

第二に、距離計算の効果がレイテンシ差を支配し、num_candidates が増えるほど悪化（WORSE）する。candidates が 40 倍に増える間（8.01 から 90.34 ms）、ES のサーバレイテンシは約 11x しか上昇しないのに対し、OpenSearch は約 18x 上昇し、OpenSearch のスループットは 14x 崩壊する（279.58 から 19.53 q/s）のに対し ES は 3.3x（534.75 から 163.31 q/s）にとどまる。candidate あたりの距離計算は訪問ノード数に比例してスケールする項であるため、より安価でキャッシュ親和性の高いカーネルを持つエンジンが、検索が重くなるほど引き離す。正直な caveat としては、ここで報告された各行は両側ともクエリレベルの rescore/oversample = 1 を用いており、OpenSearch はより弱いデフォルト（DEFAULT）の対称 1-bit パスで実行された（recall 差の大部分を埋めたであろう非対称距離計算（Asymmetric Distance Computation）は 3.x で利用可能だったが有効化されなかった）。したがって recall の差は、構成されたままのデフォルト比較を反映したものであり、アルゴリズム上の上限ではない。

04結果の読み解き

曲線を 3 つの軸で読む。(1) recall 対 num_candidates: 両者とも凹型で、速く上昇した後、共有の上限（約 0.984）に向かって平坦化する。ES はどこでも上側の包絡線である。曲線間の垂直方向の差は単調に縮小する（nc=250 で 6.81 pp、nc=1000 で 4.08 pp、nc=5000 で 0.71 pp、nc=10000 で 0.08 pp）。これはより優れた量子化推定の教科書的な特徴である（candidate 予算が乏しいところに優位が集中し、プールが網羅的になると消滅する）。(2) サーバレイテンシ 対 num_candidates: 両者とも超線形に上昇するが、OpenSearch の傾きははるかに急である。ES は 8.01 -> 13.56 -> 50.58 -> 90.34 ms と進む。OpenSearch は 45.07 -> 62.93 -> 252.93 -> 810.91 ms と進む。レイテンシ比は U 字から J 字の形状である。nc=250 で 5.6x、nc=1000 で 4.6x まで下がり（ここでは ES のスループットがまだライン速度の約 534 q/s で飽和している）、その後 candidate あたりの距離項が支配的になると 5.0x と 9.0x に上昇する。(3) スループット 対 num_candidates: ES は plateau を維持する（約 534, 534, その後 270.91 と 163.31 q/s へ低下）。nc=1000 までの平坦な領域は、ES がそこでまだ距離カーネル上で CPU バウンドになっていないことを示す。OpenSearch は最初の点から低下する（279.58 -> 19.53 q/s）。これは candidate あたりのコストが nc=250 でさえ支配的であることを意味する。クロスオーバーの洞察として、このスイープでは OpenSearch がレイテンシでも recall でも ES に並ぶ num_candidates は存在しない。ES はパレートフロンティア全体を支配する。最も運用上関連性の高い読み方は同一 recall 時である。約 0.915 の recall に到達するために ES は約 13.6 ms、OpenSearch は約 78 ms を必要とする（5.7x）。約 0.97 に到達するために ES は約 50.6 ms、OpenSearch は約 321 ms を必要とする（6.3x）。OpenSearch がほぼ並ぶ唯一の領域は nc=10000 での recall（0.984 対 0.985）だが、そこに到達するために 90 ms に対して 811 ms を支払う。

05なぜElasticsearch 9.3が勝つのか

#1非対称な int4 クエリ量子化 対 対称 1-bit が ES により正確なランキング信号を与える（低 num_candidates での recall リードを駆動する）

recall 差は nc=250 で 6.81 pp（0.7704 対 0.7023）、nc=1000 で 4.08 pp、nc=5000 で 0.71 pp、nc=10000 で 0.08 pp である。このパリティへの単調な縮小は、グラフトポロジーではなく推定品質の診断的な特徴である。BBQ はドキュメントを 1 ビット/次元で格納するが、クエリ（QUERY）は int4（4 ビット/次元）に保ち、ベクトルあたり約 14 バイトの補正係数（重心までの距離、および RaBitQ 推定器のバイアスを補正する v_c.v_bar 射影）を持つ。これらにより BBQ は真値に近いコサイン/内積スコアを代数的に再構成でき、単一の符号ビットが破壊する大きさ情報を保持する。OpenSearch 3.5 のデフォルトの FAISS 32x パスは対称であり、クエリとドキュメントの両方が +/-1 に縮約されるため、距離は本質的に 4x ハミング距離となり、本来近い点同士が同点に潰れてしまう（「オレンジ色のクエリが等距離に見える」という失敗モード）。同点の多い歪んだランキングは一部の真の近傍をカットラインの下に押しやるため、OpenSearch はそれらを回収するためにより大きな candidate プールを必要とする。num_candidates が 10000 に達すると、両方のプールが十分に大きくなり、推定の鋭さに関わらず真の上位 100 件が浮上するため、recall は両側とも約 0.984 に収束する。これはまさにデータが示す通りである。

#2より安価な candidate あたりの距離カーネルにより、ES のレイテンシ優位が num_candidates とともに拡大（WIDEN）する（5.6x -> 9.0x）

ES が支配的なのは recall だけでなくサーバレイテンシであり、その差は検索の労力とともに拡大する。num_candidates が 250->10000（40 倍の candidates）と増えるとき、ES のサーバレイテンシは 11.3x（8.01->90.34 ms）上昇するが、OpenSearch は 18.0x（45.07->810.91 ms）上昇する。スループットは ES が 3.3x、OpenSearch が 14.3x 崩壊する。労力とともにスケールするコスト項は、HNSW 巡回中に実行される距離計算の数（訪問ノード数に比例し、これはビーム幅 = num_candidates とともに増大する）である。BBQ の距離は、1-bit のドキュメントコードに対する 4 つの int4 クエリビットプレーンとのビット演算 AND + popcount であり、Lucene の Panama パスによって AVX VPOPCNTQ（スカラー比 8x-30x 高速）にコンパイルされる。OpenSearch の FAISS 32x の第一段階距離も高速なハミング popcount だが、呼び出しごとに JNI 境界を越えて到達される。そして決定的に、OpenSearch は ef_search = num_candidates に設定するため、nc=10000 ではシャードあたり 10000 幅のビームを探索することになる。実効ビーム幅がより広いことと、比較あたりの定数オーバーヘッドがより高いことの組み合わせが、OpenSearch のレイテンシが超線形にスケールし、ES が劣線形にスケールする理由である。nc=250 ではカーネルコストが小さく比率は 5.6x だが、nc=10000 までに candidate あたりの項が支配的となり、比率は 9.0x になる。

#3より低い num_candidates でより高い recall が得られることでレイテンシの勝利が複合する（同一 candidates ではなく同一 recall 時）

ES はより高い recall 対 candidates 曲線上に位置するため、固定（FIXED）された recall 目標での公平な比較は、num_candidates をマッチさせた各行よりもさらに一方的になる。約 0.915 の recall に到達するために、ES は nc=1000（13.56 ms）を必要とするのに対し、OpenSearch は 0.90 を超えるためにおおよそ nc=1500（77.96 ms）を必要とする。これは同一 recall でのレイテンシ比 5.7x である。約 0.97-0.975 に到達するために、ES は nc=5000（50.58 ms）を必要とするのに対し、OpenSearch は約 nc=6000（320.59 ms）を必要とする。これは 6.3x の比率である。したがって、ランキング品質の優位（factor 1）とカーネルコストの優位（factor 2）が乗算される。ES はより少ない candidates で目標 recall に到達し、しかも各 candidate をより安価に処理する。

#4構成されたままのデフォルトが recall 軸で ES に有利だった（公平性に関する factor）

これらの行は両エンジンともクエリレベルの rescore = 1（s_n_r の '_1' サフィックス）を用いており、OpenSearch はそのデフォルト（DEFAULT）の対称 1-bit エンコーダで in_memory FAISS 32x として実行された。OpenSearch の非対称距離計算（ADC）は、クエリをフルプレシジョンに保ち、recall 差を最も直接的に埋めるであろう単一の機能だが、これはオプトイン（enable_adc:true）であり設定されていなかった。したがって recall 差の一部は、OpenSearch にとってのアルゴリズム上の硬い上限ではなく、デフォルト対デフォルトの産物である。BBQ はその非対称性をデフォルトでオンの状態で出荷するが、OpenSearch はそうしなかった。これはレイテンシ差には影響しない（それは構造的なものである）が、recall 曲線に関する主張には但し書きが付く。

06技術ディープダイブ

Elasticsearch Better Binary Quantization（dense_vector index_options type=bbq_hnsw）
仕組み

各 Lucene セグメントは単一の重心を計算する。ドキュメントベクトルは中心化（v-c）され、符号によって 1 ビット/次元に量子化され、32x の圧縮に加えてベクトルあたり約 14 バイトの補正係数（重心までの距離、および RaBitQ 推定器のバイアスを補正する、中心化ベクトルの量子化形式への射影）が得られる。クエリは非対称に扱われる。同じ重心で中心化され、int4（4 ビット/次元）に量子化され、決して 1 ビットにはされず、クエリは一過性であるためストレージコストはゼロである。距離は、int4 クエリを 4 つの位置ビットプレーンにビットスライスし、(bitCount(d&q1)<<0)+(bitCount(d&q2)<<1)+(bitCount(d&q3)<<2)+(bitCount(d&q4)<<3) を計算し、格納された係数で実際のコサイン/内積スコアに補正することで推定される。8.18/9.x の時点で、背後のアルゴリズムは Optimized Scalar Quantization（ベクトルごとの分位点）であり、これが recall をさらに高める。

この結果への影響

これは、同一（SAME）num_candidates における ES のより高い recall（nc=250 で 0.7704 対 0.7023）の根本原因である。int4 クエリ + 補正係数は、対称な符号ビット比較が捨ててしまう大きさ情報を保持するため、真の近傍が第一段階の candidate リストでより上位にランクされ、より小さなプールに出現する。また、これが nc=10000 で recall 差が 0.08 pp まで崩壊する理由でもある。より鋭い推定は、candidate 予算がタイトなときに最も重要となるのである。

BBQ ビット演算距離カーネル + Lucene Panama SIMD（AVX VPOPCNTQ / NEON CNT）
仕組み

ビットベクトル内積は、論理 AND の後に population count を行い、LongVector に累積して reduce することで実装される。4 つのクエリビットプレーンは独立したアキュムレータにループアンロールされ、ドキュメントコードの再ロードを回避する。x86 ではこれは vpand/vpopcntq にコンパイルされ、ARM では NEON AND/CNT/UADDLP にコンパイルされる。スカラーに対する測定された高速化は、384 次元で 8x、1536 次元では 30x まで上昇する。

この結果への影響

これはレイテンシ曲線の背後にある荷重を支える（load-bearing）factor である。距離カーネルは、訪問ノード数（= ビーム幅 = num_candidates）とともにスケールする candidate あたりのコストである。このカーネルが非常に安価であるため、candidates が 40 倍に増えても ES のサーバレイテンシは 11.3x しか上昇せず、スループットは nc=1000 まで約 534 q/s 付近に張り付き、nc=10000 までに 3.3x 低下するにとどまる。128 次元ではカーネルは絶対的には小さく、これがまさに ES が nc=250 で約 8 ms を維持する理由である。

OpenSearch FAISS エンジン、compression_level=32x（1-bit バイナリ量子化）、mode=in_memory
仕組み

compression_level=32x は各 float32 次元を単一ビットにマップする（32x 削減）。これらのバイナリコード上の FAISS HNSW インデックスはオフヒープのネイティブメモリに存在する。第一段階距離はバイナリコード上の対称ハミング距離（xor + popcount）であり、C++ で計算され、JNI を介して JVM から呼び出される。oversample_factor=1.0 の in_memory モードでは、実質的にフルプレシジョンのリスコア展開は行われず、recall は量子化されたランキングと HNSW ビームに依存する。このベンチマークのクエリは ef_search = num_candidates に設定されるため、最下層の candidate リストはスイープパラメータと直接スケールする。

この結果への影響

帰結は 2 つある。(a) 対称 1-bit 量子化はクエリも量子化するため大きさ情報を失い、同点の多いランキングを生み出す。これが OpenSearch が同じ recall のためにより多くの candidates を必要とする理由である（nc=250 で 0.7023 対 0.7704）。(b) ef_search=num_candidates に設定することは、nc=10000 で OpenSearch がシャードあたり 10000 幅のビームを巡回することを意味する。ES の JVM 内 SIMD カーネルよりも比較あたりの定数コストが高いため、これがレイテンシが 810.91 ms まで超線形にスケールし、スループットが 19.53 q/s まで崩壊する理由である。

m=16, ef_construction=100 での HNSW グラフ巡回（両側で同一）
仕組み

両エンジンは同じ navigable small-world グラフを構築する。最大次数 m=16、構築時の ef_construction=100 である。クエリ時には、ビーム幅（ES では num_candidates、OS では ef_search=num_candidates）が貪欲降下の間にどれだけの candidates を生存させるかを制御する。candidates が多いほど、近傍リストの参照外しと距離計算が多くなり、recall が高くなりレイテンシも高くなる。

この結果への影響

グラフパラメータが同一であるため、HNSW トポロジーはここでの差別化要因ではない（NOT）。それは制御された変数である。README の「より効率的なグラフ巡回（more efficient graph traversal）」という表現は不正確である。グラフは同じ形状を持つ。真のレバーは、量子化推定の品質（同じ巡回がどの近傍を浮上させるか）と、距離あたりのカーネルコスト（各ホップがどれだけ速いか）である。これが、本分析がこの結果をグラフ構造ではなく BBQ の非対称性 + SIMD カーネルに帰する理由である。

2 段階のリスコア / オーバーサンプル（ES では rescore_vector.oversample、OS では rescore.oversample_factor）
仕組み

両エンジンとも、量子化インデックスから O*k 個の candidates を集め、上位 k 件を生の float32 ベクトルで再ランキングすることをサポートする。ES はマッピングレベルの BBQ デフォルトオーバーサンプル 3.0 を設定するが、報告された行はクエリレベルのオーバーサンプル = 1 を渡す。OpenSearch は oversample_factor = 1 を渡す。oversample=1 でも、ES は依然として上位 k 件を float32 に対してリスコアする（oversample=0 だと無効化される）が、OpenSearch の in_memory モードはリスコア展開を一切適用しない。

この結果への影響

決定的なことに、ヘッドラインの行は ES の recall リードを作り出すために 3x のフルプレシジョンリスコアに依存していない（NOT）。両側ともオーバーサンプルのつまみを 1 で実行する。これにより比較が量子化ランキングの品質 + HNSW に限定され、BBQ の非対称推定への帰属が強化される。これはまた重要な公平性の caveat でもある。異なるリスコア設定（または OpenSearch ADC の有効化）は recall 曲線を移動させるであろう。

07留意点・公平性

  • 報告された行は両（BOTH）エンジンともクエリレベルの rescore/oversample = 1 を用いているため、recall リードは 3x のフルプレシジョンリスコアによって生み出されたものではない。これは量子化品質を切り分けるには好都合だが、ES の BBQ マッピングはデフォルトのオーバーサンプル 3.0 を持ち、クエリがそれを 1 に上書きしている点に留意せよ。異なるオーバーサンプル設定は両方の曲線を動かすであろう。
  • OpenSearch 3.5 はそのデフォルト（DEFAULT）の対称 1-bit FAISS パスで実行された。バイナリ距離推定を最も直接的に改善し recall 差を縮小したであろう機能である非対称距離計算（ADC、enable_adc:true）は、OpenSearch 3.x で利用可能だったが有効化されなかった。したがって recall 差はデフォルト対デフォルトの比較であり、OpenSearch にとってのアルゴリズム上の上限ではない。
  • バージョンの交絡: OpenSearch 3.6（このベンチマークの後）はデフォルトの 32x FAISS パスを 1-bit Scalar Quantization に変更し、古いバイナリ手法に対して約 24% 良い recall と約 15% 低いレイテンシを主張している。3.6 で再実行すれば、recall 差と（ある程度）レイテンシ差の両方がおそらく縮小するであろう。
  • ES bbq_hnsw と OS FAISS 32x はどちらも 1-bit/次元だが、エンジンの実行パスは異なる。ES は（9.3 時代では）Lucene Panama を介して JVM 内で SIMD 距離カーネルを実行するのに対し、OpenSearch は JNI 境界を越えてネイティブ FAISS に入る。candidate あたりのコスト差の一部は、純粋に量子化の数学だけではなく、このアーキテクチャの非対称性である。（注: ES 9.4 では Elasticsearch もホットカーネルを Panama FFI 経由でネイティブに移行したため、この特定の対比はバージョンに依存する。）
  • ef_search のセマンティクスは微妙に異なる。ES は k と num_candidates の両方を同じ値にスイープする。OpenSearch は k をスイープし ef_search=num_candidates を設定する。意図は比較可能なビーム幅だが、2 つのシステムは candidate リスト/早期終了のセマンティクスを異なって定義しているため、「同じ num_candidates」は検索労力の近似的な等価であって厳密なものではない。
  • データセットはわずか 128 次元である。BBQ のスカラーに対する SIMD 高速化は次元数とともに増大する（384 次元で 8x、1536 次元で 30x）ため、128 次元ではカーネル優位はその範囲の低（LOW）域にある。より高次元の埋め込みでは、レイテンシ差はさらに大きくなる可能性がある。逆に、補正係数のオーバーヘッドは 128 次元では相対的により大きな割合を占める。
  • README はこの勝利を「より効率的なグラフ巡回（more efficient graph traversal）」に帰しているが、これは不正確である。両エンジンとも同一の HNSW パラメータ（m=16, ef_construction=100）を使用している。巡回アルゴリズムは制御された変数であり、真の原因は量子化推定の品質と距離カーネルのコストである。
  • ここで分析されているのはサーバ側レイテンシ（server_latency_avg）のみである。エンドツーエンドのクライアントレイテンシ（latency_avg）はネットワーク/シリアライゼーションを含み、同じ順序を示すが絶対値は異なる。スループットは単一ストリームから導出されたものであり、（ネイティブのオフヒープメモリと GC 圧力がエンジン間で異なる）並行性の挙動は、このスイープでは検証されていない。

08結論

Elasticsearch 9.3 BBQ HNSW は、このベンチマークのパレートフロンティア全体にわたって OpenSearch 3.5 FAISS 32x を、分離可能でメカニズム的に異なる 2 つの理由により支配する。recall の優位（低 num_candidates で最大の 6.8 pp、nc=10000 で 0.08 pp まで消失）は、BBQ の非対称な int4 クエリ / 1-bit ドキュメント量子化と補正係数に由来し、これらはクエリの大きさを同点に失う OpenSearch のデフォルト対称 1-bit バイナリ量子化よりもはるかに正確な距離推定を再構成する。レイテンシの優位（5.6x から 9.0x、検索労力とともに拡大）は、BBQ の安価なビット演算 AND+popcount SIMD カーネルが candidate あたりの距離コストを低く保つことに由来する一方、OpenSearch のより広いビーム（ef_search=num_candidates）と比較あたりの高いオーバーヘッドが、そのレイテンシを超線形にスケールさせ、スループットを 14x 崩壊させる。これらは同一 recall 時に複合する（マッチした recall でレイテンシ比は約 6x）。この結果は本物で一貫しているが、それは構成されたままのデフォルト比較である。すなわち、ES が非対称量子化をデフォルトでオンにして出荷している一方、OpenSearch 3.5 のより強力な ADC パスとその 3.6 の SQ デフォルトが関与していなかったことを反映している。正直な枠組みは、Elasticsearch の BBQ デフォルトがこの次元数において recall におけるレイテンシ（latency-at-recall）の目標に対して例外的によくチューニングされており、このベンチマークはどちらのエンジンの絶対的なアルゴリズム限界ではなく、そのデフォルト品質を測定しているということである。

この分析の作り方: 数値はすべて本リポジトリの実測結果 CSV に基づきます。「なぜ」の技術解説は、複数エージェントによる一次資料（Elastic search-labs / Lucene / OpenSearch / Qdrant の公式ドキュメント・ブログ）の並列 Web 調査と、主要主張に対する敵対的事実検証（反証を試みる立場でのファクトチェック）を経て統合したものです。検証で訂正された点（例: DiskBBQ は HNSW ではなく IVF 系クラスタ構造、Qdrant 側は TurboQuant であり README の「binary quantization」表記は不正確、など）は本文・留意点に反映済みです。
